css世界笔记

border

border-width：
thin: 薄薄的，等同于1px;
medium: 默认值，等同于3px; （border-style:double至少3px才有效果）
thick:厚厚的，等同于4px；

margin

margin百分比值无论是水平方向还是垂直方向均是相对于宽度计算的！

margin合并

只发生在垂直方向上

  • margin-top合并：
  • margin-bottom合并

margin合并规则

总结概括为“正正取大值”“正负值相加”“负负最负值”

正正取大值

  • 如果是相邻元素合并，间距取50px 大值
  • 如果是父子合并，间距取50px 相当于father margin-top: 50px
  • 如果是自身合并，元素外部尺寸为50px 大值 【空元素】

正负相加

  • 如果是相邻元素合并，间距取30px （-20 + 50）
  • 如果是父子合并，间距取-30px （-20 + 50） 相当于father margin-top: -30px
  • 如果是自身合并，元素外部尺寸为-30px （-20 + 50）

负负取最负

  • 如果是相邻元素合并，间距取-50px 最负
  • 如果是父子合并，间距取-50px 相当于father margin-top: -50px
  • 如果是自身合并，元素外部尺寸为-50px

    margin：auto填充规则

    1、如果一侧有auto，一侧为定值，则auto为自动填充剩余空间
    2、如果两侧均为auto，则平分剩余空间

margin垂直居中：

margin失效

  • margin合并的时候，更改margin值可能是没有效果 （更改小值）

  • 绝对定位元素非定位方位的margin值“无效”

  • 定高容器的子元素的margin-bottom或者宽度定死的子元素的margin-right的定位“失效”,(影响后续兄弟元素)

  • 鞭长莫及导致margin无效

  • 内联特性导致margin无效

padding

padding 足够大时，width失效
padding百分比值无论是水平方向还是垂直方向均是相对于宽度计算的！

content 计数器

  • counter-reset 创建或者重置计数器
  • counter-increment 递增变量
  • counter()或者counters()函数 得出计数器的值

要想实现嵌套，必须让每一个列表容器拥有一个“普照源”，通过子辈对父辈的counter-reset重置、配合counters()方法才能实现计数嵌套效果

伪元素

常见伪元素 ::first-letter,::first-line,::before,::after,::selection

::before和::after必须配合content属性来使用，content用来定义插入的内容，content必须有值，至少是空。 当插入元素后content的宽度或者高度会发生变化

clear

指定一个元素是否必须移动(清除浮动后)到在它之前的浮动元素下面。clear 属性适用于浮动和非浮动元素

  • none 元素不会向下移动清除之前的浮动
  • left 元素被向下移动用于清除之前的左浮动。
  • right 元素被向下移动用于清除之前的右浮动。
  • both 元素被向下移动用于清除之前的左右浮动。
  • inline-start 该关键字表示该元素向下移动以清除其包含块的起始侧上的浮动。即在某个区域的左侧浮动或右侧浮动。
  • inline-end 该关键字表示该元素向下移动以清除其包含块的末端的浮点，即在某个区域的右侧浮动或左侧浮动。

box-sizing

盒子模型

  • box-sizing: content-box 默认值 width = content区域宽度 / height = content区域高度
  • box-sizing:border-box width = content区域宽度 + padding + border / height = content区域高度 + padding + border
box-sizing:border-box 可以使原生普通文本框```<input>```和文本域```<textarea>```100%自适应父容器宽度

http缓存

发表于 2022-01-04 分类于 前端

先说结论

  • 1、浏览器缓存分为 强缓存和协商缓存
  • 2、第一次请求时，浏览器缓将请求结果和缓存标识存入浏览器
  • 3、再起发起，浏览将在缓存中查找是否有该请求的缓存，如果命中强缓存则使用强缓存，如果没有命中，则使用协议缓存
  • 4、如果协议缓存有效，则使用协议缓存；如果失效则重新向server发起请求

强制缓存

强制缓存的字段分别是Expires和Cache-Control，其中Cache-Control优先级比Expires高

Expires

该字段是 http1.0 时的规范，它的值为一个绝对时间的 GMT 格式的时间字符串，比如 Expires:Mon,18 Oct 2066 23:59:59 GMT。

这个时间代表着这个资源的失效时间，在此时间之前，即命中缓存

该时间是一个绝对时间，如果客户端的时间和服务器的时间相差比较大，则会导致缓存机制混乱

Cache-Control

该字段是http1.1 新增的规范，他是一个相对时间；主要字段有如下几个值：

  • 1、max-age：资源有效期多长，以秒为单位 max-age=3600 表示资源可以缓存3600秒
  • 2、public：所有内容都将被缓存（客户端和代理服务器都可缓存）
  • 3、private 所有内容只有客户端可以缓存【默认值】
  • 4、no-cache 客户端缓存内容，但是是否使用缓存则需要经过协商缓存来验证决定【注意和no-store的区别】
  • 5、no-store 所有内容都不会被缓存，即不使用强制缓存，也不使用协商缓存

协商缓存

协商缓存就是强制缓存失效后，浏览器携带缓存标识向服务器发起请求，由服务器根据缓存标识决定是否使用缓存的过程，主要有以下两种情况

  • 1、如果协商缓存生效，返回304
  • 2、协商缓存失效，则返回200及新的资源

etag / if-none-match优先级高于last-modified / if-modified-since，同时存在则只有etag / if-none-match生效

etag / if-none-match

etag是服务器响应请求时，返回当前资源文件的一个唯一标识(由服务器生成)

if-none-match是客户端再次发起该请求时，请求头上携带上次请求返回的唯一标识etag值。
服务器接收到请求后，发现请求头含有if-none-match,则服务器会根据携带的etag字段值跟服务器上资源的etag比对，如果一致，则返回304状态，说明协商缓存生效；如果不一致，则重新返回资源，状态为200。

nginx可以简单的配置是否启用etag验证

last-modified / if-modified-since

last-modified是请求返回时携带的资源文件的修改时间
if-modified-since是客户端再次发起该请求时，请求头上携带上次请求返回的last-modified时间，
服务器接收到请求后，发现请求头含有if-modified-since,则服务器会根据携带字段值跟服务器上资源的最后修改时间比对，如果文件最后修改的时间大于if-modified-since的字段值，重新返回资源，状态为200；否则返回304状态，说明协商缓存生效。
nginx可以简单的配置是否启用last-modified验证

memory cache 和 disk cache

浏览器中，浏览器会在js和图片等文件解析执行后直接存入内存缓存中，那么当刷新页面时只需直接从内存缓存中读取(from memory cache)；而css文件则会存入硬盘文件中，所以每次渲染页面都需要从硬盘读取缓存(from disk cache)。

  • 内存缓存在缓存资源时并不关心返回资源的HTTP缓存头Cache-Control是什么值，同时资源的匹配也并非仅仅是对URL做匹配，还可能会对Content-Type，CORS等其他特征做校验。
  • memory cache 一旦该进程关闭，则被清空
  • disk cache 会根据 HTTP Herder 中的字段判断哪些资源需要缓存，哪些资源可以不请求直接使用，哪些资源已经过期需要重新请求
  • 因为CSS文件加载一次就可渲染出来,我们不会频繁读取它,所以它不适合缓存到内存中,但是js之类的脚本却随时可能会执行,如果脚本在磁盘当中,我们在执行脚本的时候需要从磁盘取到内存中来,这样IO开销就很大了,有可能导致浏览器失去响应

浏览器会把哪些文件丢进内存中？哪些丢进硬盘中？网上说法不一，不过以下观点比较靠得住:

  • 对于大文件来说，大概率是不存储在内存中的，反之优先
  • 当前系统内存使用率高的话，文件优先存储进硬盘

F5及Ctrl+F5

  • 地址栏访问 用户正常访问，触发浏览器缓存机制
  • F5 普通刷新，chrome上操作一下和地址栏点击回车访问一致，此外，因为浏览器标签没有关闭，进程还存在，则memory cache 和 disk cache是可用的
  • Ctrl+F5 强制刷新，浏览器不使用缓存，
    • 1、 请求的头部会携带Cache-control:no-cache(为了兼容，还带了 Pragma:no-cache)设置强缓存策略，而且也不会携带 if-none-match 或者 if-modified-since不启用协商缓存，所以会得到最新的server资源

css高度百分百与100vh的区别

发表于 2021-12-31 更新于 2022-01-06 分类于 css

vh: 相对于视窗的高度, 视窗被均分为100单位的vh
正常情况下

height:100vh == height:100%;

注意点：
100vh：是相对视窗的高度
100%：是相对父元素的高度，如果父元素高度为0，即使子元素设置了height:100%,其高度依旧为0

height:100%
绝对定位元素的百分比计算和非绝对定位元素的百分比计算是有区别的，区别在于绝对定位的宽高百分比计算是相对于padding box的，也就是说会把padding大小值计算在内，但是，非绝对定位元素则是相对于content box计算的

多个commit合并

发表于 2021-12-31 分类于 git

初始化环境

合并

  • 目的： 将【第三次提交】【第二次提交】【第一次提交】合并到【要合并的位置】并重新提交
  • 通过git log查看提交的日志
  • 使用rebase命令设置提交的位置，这里有个需要注意的地方，因为我想把最后的三次提交合到要合并的位置，这里其实是需要合并最后的四次提交，所以需要合并的commit为5865c72e6d8ba0bbd6c53eac6de7a865f0b571cf及【要合并的位置】的上一次提交
  • 使用命令git rebase -i 5865c7或者git rebase -i HEAD~4 会弹出一个执行命令的编辑界面
  • 我们将 最后的三次commit 前方的命令改成 squash 或 s，然后输入:wq以保存并退出
    • pick 的意思是要会执行这个 commit
    • squash 的意思是这个 commit 会被合并到前一个commit
  • 保存之后我们会看到 commit message 的编辑界面
  • 要做的就是将多个commit信息合并为一个
  • 这个时候再使用git log或者可视化工具查看就会发现多次提交已经合并，并且合并后的commit也已经修改
