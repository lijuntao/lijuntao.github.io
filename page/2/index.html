border

border-width：
thin: 薄薄的，等同于1px;
medium: 默认值，等同于3px; （border-style:double至少3px才有效果）
thick:厚厚的，等同于4px；

margin

margin百分比值无论是水平方向还是垂直方向均是相对于宽度计算的！

margin合并

只发生在垂直方向上

  • margin-top合并：
  • margin-bottom合并

margin合并规则

总结概括为“正正取大值”“正负值相加”“负负最负值”

正正取大值

  • 如果是相邻元素合并，间距取50px 大值
  • 如果是父子合并，间距取50px 相当于father margin-top: 50px
  • 如果是自身合并，元素外部尺寸为50px 大值 【空元素】

正负相加

  • 如果是相邻元素合并，间距取30px （-20 + 50）
  • 如果是父子合并，间距取-30px （-20 + 50） 相当于father margin-top: -30px
  • 如果是自身合并，元素外部尺寸为-30px （-20 + 50）

负负取最负

  • 如果是相邻元素合并，间距取-50px 最负
  • 如果是父子合并，间距取-50px 相当于father margin-top: -50px
  • 如果是自身合并，元素外部尺寸为-50px

    margin：auto填充规则

    1、如果一侧有auto，一侧为定值，则auto为自动填充剩余空间
    2、如果两侧均为auto，则平分剩余空间

margin垂直居中：

margin失效

  • margin合并的时候，更改margin值可能是没有效果 （更改小值）

  • 绝对定位元素非定位方位的margin值“无效”

  • 定高容器的子元素的margin-bottom或者宽度定死的子元素的margin-right的定位“失效”,(影响后续兄弟元素)

  • 鞭长莫及导致margin无效

  • 内联特性导致margin无效

padding百分比值无论是水平方向还是垂直方向均是相对于宽度计算的！

指定一个元素是否必须移动(清除浮动后)到在它之前的浮动元素下面。clear 属性适用于浮动和非浮动元素

  • inline-end 该关键字表示该元素向下移动以清除其包含块的末端的浮点，即在某个区域的右侧浮动或左侧浮动。

http缓存

发表于 2022-01-04 分类于 前端

先说结论

  • 1、浏览器缓存分为 强缓存和协商缓存
  • 2、第一次请求时，浏览器缓将请求结果和缓存标识存入浏览器
  • 3、再起发起，浏览将在缓存中查找是否有该请求的缓存，如果命中强缓存则使用强缓存，如果没有命中，则使用协议缓存
  • 4、如果协议缓存有效，则使用协议缓存；如果失效则重新向server发起请求

强制缓存

强制缓存的字段分别是Expires和Cache-Control，其中Cache-Control优先级比Expires高

Expires

该字段是 http1.0 时的规范，它的值为一个绝对时间的 GMT 格式的时间字符串，比如 Expires:Mon,18 Oct 2066 23:59:59 GMT。

这个时间代表着这个资源的失效时间，在此时间之前，即命中缓存

该时间是一个绝对时间，如果客户端的时间和服务器的时间相差比较大，则会导致缓存机制混乱

Cache-Control

该字段是http1.1 新增的规范，他是一个相对时间；主要字段有如下几个值：

  • 1、max-age：资源有效期多长，以秒为单位 max-age=3600 表示资源可以缓存3600秒
  • 2、public：所有内容都将被缓存（客户端和代理服务器都可缓存）
  • 3、private 所有内容只有客户端可以缓存【默认值】
  • 4、no-cache 客户端缓存内容，但是是否使用缓存则需要经过协商缓存来验证决定【注意和no-store的区别】
  • 5、no-store 所有内容都不会被缓存，即不使用强制缓存，也不使用协商缓存

协商缓存

协商缓存就是强制缓存失效后，浏览器携带缓存标识向服务器发起请求，由服务器根据缓存标识决定是否使用缓存的过程，主要有以下两种情况

  • 1、如果协商缓存生效，返回304
  • 2、协商缓存失效，则返回200及新的资源

etag / if-none-match优先级高于last-modified / if-modified-since，同时存在则只有etag / if-none-match生效

etag / if-none-match

etag是服务器响应请求时，返回当前资源文件的一个唯一标识(由服务器生成)

if-none-match是客户端再次发起该请求时，请求头上携带上次请求返回的唯一标识etag值。
服务器接收到请求后，发现请求头含有if-none-match,则服务器会根据携带的etag字段值跟服务器上资源的etag比对，如果一致，则返回304状态，说明协商缓存生效；如果不一致，则重新返回资源，状态为200。

nginx可以简单的配置是否启用etag验证

last-modified / if-modified-since

last-modified是请求返回时携带的资源文件的修改时间
if-modified-since是客户端再次发起该请求时，请求头上携带上次请求返回的last-modified时间，
服务器接收到请求后，发现请求头含有if-modified-since,则服务器会根据携带字段值跟服务器上资源的最后修改时间比对，如果文件最后修改的时间大于if-modified-since的字段值，重新返回资源，状态为200；否则返回304状态，说明协商缓存生效。
nginx可以简单的配置是否启用last-modified验证

memory cache 和 disk cache

浏览器中，浏览器会在js和图片等文件解析执行后直接存入内存缓存中，那么当刷新页面时只需直接从内存缓存中读取(from memory cache)；而css文件则会存入硬盘文件中，所以每次渲染页面都需要从硬盘读取缓存(from disk cache)。

  • 内存缓存在缓存资源时并不关心返回资源的HTTP缓存头Cache-Control是什么值，同时资源的匹配也并非仅仅是对URL做匹配，还可能会对Content-Type，CORS等其他特征做校验。
  • memory cache 一旦该进程关闭，则被清空
  • disk cache 会根据 HTTP Herder 中的字段判断哪些资源需要缓存，哪些资源可以不请求直接使用，哪些资源已经过期需要重新请求
  • 因为CSS文件加载一次就可渲染出来,我们不会频繁读取它,所以它不适合缓存到内存中,但是js之类的脚本却随时可能会执行,如果脚本在磁盘当中,我们在执行脚本的时候需要从磁盘取到内存中来,这样IO开销就很大了,有可能导致浏览器失去响应

浏览器会把哪些文件丢进内存中？哪些丢进硬盘中？网上说法不一，不过以下观点比较靠得住:

  • 对于大文件来说，大概率是不存储在内存中的，反之优先
  • 当前系统内存使用率高的话，文件优先存储进硬盘

F5及Ctrl+F5

  • 地址栏访问 用户正常访问，触发浏览器缓存机制
  • F5 普通刷新，chrome上操作一下和地址栏点击回车访问一致，此外，因为浏览器标签没有关闭，进程还存在，则memory cache 和 disk cache是可用的
  • Ctrl+F5 强制刷新，浏览器不使用缓存，
    • 1、 请求的头部会携带Cache-control:no-cache(为了兼容，还带了 Pragma:no-cache)设置强缓存策略，而且也不会携带 if-none-match 或者 if-modified-since不启用协商缓存，所以会得到最新的server资源

vh: 相对于视窗的高度, 视窗被均分为100单位的vh
正常情况下

height:100vh == height:100%;

注意点：
100vh：是相对视窗的高度
100%：是相对父元素的高度，如果父元素高度为0，即使子元素设置了height:100%,其高度依旧为0

height:100%
绝对定位元素的百分比计算和非绝对定位元素的百分比计算是有区别的，区别在于绝对定位的宽高百分比计算是相对于padding box的，也就是说会把padding大小值计算在内，但是，非绝对定位元素则是相对于content box计算的

Electron基本概念整理

详细参考文档

官方文档

主线程 main process

主进程，通常是名为main.js 的文件，是每个 Electron 应用的入口文件。它控制着整个 App 的生命周期，从打开到关闭。 它也管理着系统原生元素比如菜单，菜单栏，Dock 栏，托盘等。 主进程负责创建 APP 的每个渲染进程。而且整个 Node API 都集成在里面

Electron 运行 package.json 的 main 脚本的进程被称为主进程。 在主进程中运行的脚本通过创建web页面来展示用户界面。 一个 Electron 应用总是有且只有一个主进程。

渲染进程

渲染进程是你的应用内的一个浏览器窗口。与主进程不同的是，它能够同时存在多个而且运行在不一样的进程。而且它们也能够被隐藏。

每个 Electron 中的 web 页面运行在它自己的渲染进程中。

主进程和渲染进程之间的区别

主进程使用 BrowserWindow 实例创建页面。 每个 BrowserWindow 实例都在自己的渲染进程里运行页面。 当一个 BrowserWindow 实例被销毁后，相应的渲染进程也会被终止。主进程管理所有的web页面和它们对应的渲染进程。
每个人窗口or web页面都是一个渲染进程。每个渲染进程都是独立的，它只关心它所运行的 web 页面。

进程之间通信

主进程到渲染进程的异步通信

ipcMain模块是EventEmitter类的一个实例。 当在主进程中使用时，它处理从渲染器进程（网页）发送出来的异步和同步信息。 从渲染器进程发送的消息将被发送到该模块。

渲染进程之间通信

  • 1、可以通过将消息发送至主进程，然后再目标渲染进程

  • 2、通过目标渲染进程的句柄发送消息

app

控制你的应用程序的事件生命周期。

提供了app运行时的一些事件及方法

BrowserWindow

创建和控制浏览器窗口。

BrowserView

创建和控制视图

BrowserView 被用来让 BrowserWindow 嵌入更多的 web 内容。 它就像一个子窗口，除了它的位置是相对于父窗口。 这意味着可以替代webview标签.

remote

在渲染进程中使用主进程模块。

remote 模块为渲染进程（web页面）和主进程通信（IPC）提供了一种简单方法。

在Electron中, GUI 相关的模块 (如 dialog、menu 等) 仅在主进程中可用, 在渲染进程中不可用。 为了在渲染进程中使用它们, ipc 模块是向主进程发送进程间消息所必需的。 使用 remote 模块, 你可以调用 main 进程对象的方法, 而不必显式发送进程间消息

类: WebContents

对话框

显示用于打开和保存文件、警报等的本机系统对话框。

创建OS(操作系统)桌面通知 主进程

标签

在一个独立的 frame 和进程里显示外部 web 内容。

使用 webview 标签在Electron 应用中嵌入 “外来” 内容 (如 网页)。外来”内容包含在 webview 容器中。 应用中的嵌入页面可以控制外来内容的布局和重绘。

与 iframe 不同, webview 在与应用程序不同的进程中运行。它与您的网页没有相同的权限, 应用程序和嵌入内容之间的所有交互都将是异步的。 这将保证你的应用对于嵌入的内容的安全性。

react全家桶

发表于 2021-12-20 更新于 2022-01-20 分类于 前端

react全家桶（react、redux、react-redux、react-dom、react-router、react-router-dom）

redux

redux

全局数据状态管理工具（状态管理机），用作组件通信

state

一个数据对象，存储redux中的数据。不能直接修改，只能通过action来修改

action

每一次修改数据的动作可以描述为一个action，如果需要更改state中的数据，就发送一个action。一般为 {type,payload}，其中的type属性是必须的，表示 Action 的名称。

reducer

更改state的具体实现，根据action来最state中的数据进行操作更新。
可以添加多个reducer，redux 提供了一个combineReducers方法，将各个reducer函数合成一个大的reducer。

store

全局保存数据的地方

dispatch

store.dispatch()是组件发出action的唯一方法。

react-redux

redux 的作者封装了一个 react 专用的库 react-redux
这个库是可以选用的。实际项目中，你应该权衡一下，是直接使用 Redux，还是使用 React-Redux。后者虽然提供了便利，但是需要掌握额外的 API，并且要遵守它的组件拆分规范。

UI组件

react-redux 将所有组件分成两大类：UI 组件（presentational component）和容器组件（container component）

UI组件

数据显示组件，处理界面数据显示逻辑

容器组件

记住一句话就可以了：UI 组件负责 UI 的呈现，容器组件负责管理数据和逻辑。

connect() 用户从UI组件生成容器组件

connect的意思，就是将这两种组件连起来。

完整API如下

  1. 输入逻辑：外部的数据（即state对象）如何转换为 UI 组件的参数
  2. 输出逻辑：用户发出的动作如何变为 Action 对象，从 UI 组件传出去。

mapStateToProps()

建立一个从（外部的）state对象到（UI 组件的）props对象的映射关系。

  • mapStateToProps的第一个参数总是state对象，还可以使用第二个参数，代表容器组件的props对象。

  • mapStateToProps会订阅 Store，每当state更新的时候，就会自动执行，重新计算 UI 组件的参数，从而触发 UI 组件的重新渲染。

  • connect方法可以省略mapStateToProps参数，那样的话，UI 组件就不会订阅Store，就是说 Store 的更新不会引起 UI 组件的更新。

mapDispatchToProps()

用来建立 UI 组件的参数到store.dispatch方法的映射。

  • mapDispatchToProps也有两个参数，dispatch对象，和第二个参数ownProps，代表容器组件的props对象。

react-router和react-router-dom的区别

react-router: 实现了路由的核心功能

react-router-dom: 基于react-router，加入了在浏览器运行环境下的一些功能

  • 路由组件（作为根组件）: BrowserRouter（history模式） 和 HashRouter（hash模式）保持UI和URL的同步
    区别：前者基于rul的pathname段，后者基于hash段
如果使用BrowserRouter，每次改变路由时，会向服务器发送请求。
使用HashRouter，每次改变路由只是地址栏的修改，由前端完成路由切换，服务器始终是根路由
  • 路径匹配组件: Route 和 Switch

  • Redirect 导航到一个新的地址。即重定向

  • withRouter
    可以将一个非路由组件包裹为路由组件，使这个非路由组件也能访问到当前路由的match, location, history对象。

  • 导航组件: Link 和 NavLink 可以用来指定路由跳转
